## 2026-07-12 — Knowledge Graph v1 additive foundation (issue #12)
### Added
- `docs/KNOWLEDGE_GRAPH_V1.md`, `docs/KNOWLEDGE_GRAPH_MIGRATION.md`, `docs/CURRENT_DATA_TO_GRAPH_MAPPING.md` — repository specification for an additive, backward-compatible content graph (brands, retailers, offers, products, outfits, guides, collections, and typed relationships with confidence/verification metadata), a phased migration plan, and a field-by-field audit of every `js/products.js`/`js/guides.js` field against the new entity model. **Written without access to the canonical Notion specification the issue names** (no network/Notion access in this environment) — derived instead from `CLAUDE.md`, `ARCHITECTURE.md`, `docs/CURRENT_STATE.md`, `docs/FEATURE_INVENTORY.md`, and the live data itself, with every design decision cited back to a concrete repository fact. Flagged explicitly in `docs/KNOWLEDGE_GRAPH_V1.md` for a human reviewer to reconcile against Notion before treating it as final.
- `data/schema-version.js`, `data/taxonomies.js`, `data/brands.js`, `data/retailers.js`, `data/offers.js`, `data/products.js`, `data/outfits.js`, `data/guides.js`, `data/collections.js`, `data/relationships.js` — additive schema modules. All entity records are computed programmatically from `js/products.js`/`js/guides.js` at import time (not hand-transcribed), so they cannot drift from the two files that remain canonical; `collections.js` ships with zero records since no collection concept exists in the current data (no fabricated example collections). `data/relationships.js` derives ~225 typed edges (`MADE_BY`, `HAS_OFFER`, `OFFERED_BY`, `FEATURED_IN`, `RELATED_TO`, `CONTAINS_OUTFIT`, `INCLUDES_PRODUCT`, `ALTERNATIVE_TO`) from existing legacy fields; the one `ALTERNATIVE_TO` edge in the current data (`on-cloud-x4` → `light-jeans`) is marked `confidence: "unverified"`/`verificationStatus: "draft"` because its source field is itself marked incomplete in `js/products.js`'s own TODO comment, which also correctly excludes it from `isPubliclyRecommendable()`.
- `data/adapters.js` — compatibility adapter layer reconstructing the legacy `js/products.js`/`js/guides.js` page contracts from the new `data/*.js` modules, for comparison only. **Not wired into any `.dc.html` page** — every page still imports directly from `js/products.js`/`js/guides.js`, unchanged.
- `scripts/validate-knowledge-graph.mjs` — dependency-free Node (ESM) validator, same style as `scripts/validate-content-data.mjs`: unique ids/slugs per entity type, all cross-entity references resolve, approved taxonomy/vocabulary membership, relationship predicate/confidence/verification-status validity, no duplicate normalized relationships, no self-alternatives, no circular `REPLACES` chains, and that commerce facts (`price`, `affiliateUrl`) stay `null`/`unpublished` rather than fabricated.
- `scripts/compare-legacy-adapter.mjs` — report-only (always exits 0) diff between the legacy files and `data/adapters.js`'s reconstruction of them, normalizing `""`/`[]`/omitted-key as equivalent so the one documented intentional difference class (the sparse `comingSoon` guide placeholder) doesn't produce noise.
- `.github/workflows/content-validation.yml`: new `validate-knowledge-graph` job runs both new scripts on every PR/push to `main`, alongside the existing content-data and static-site-qa jobs.
- `.github/workflows/pages.yml`: the `validate` job (which gates the Pages deploy) now also runs `scripts/validate-knowledge-graph.mjs`.
- `README.md`, `DEVELOPMENT.md`, `.github/pull_request_template.md` updated to document the two new scripts and the new `data/` directory.

- The required sanity-test (temporarily break one entity reference, confirm the validator fails, then revert) — not performed.

Every `data/*.js` module derives its records from `js/products.js`/`js/guides.js` programmatically (rather than being hand-typed) specifically to reduce the risk of an error going undetected without the ability to execute the code, and the adapter/relationship logic was manually traced against representative records (a product with a styling profile and an alternative, a product with no brand, a fully-populated guide, and the sparse `comingSoon` placeholder guide) field-by-field before this entry was written. **Run the four commands above before merging this PR** — see `.github/pull_request_template.md`'s checklist.

## 2026-07-12 — Preview 404s eliminated + static-site QA (issue #7)
### Fixed
- `guide-on-cloud-x4.dc.html`, `guide-nb9060.dc.html`, `guide-barrel-pants-nb530.dc.html`: the hero cover image's `loading="eager"` caused the browser to eagerly fetch the raw, unhydrated `<img src="{{ coverImage }}">` template token as a literal URL during the browser's initial native HTML parse (before the dc-runtime replaced it) — every guide page load produced one harmless-but-visible `404` for `/%7B%7B%20coverImage%20%7D%7D`. Changed to `loading="lazy"`, which lets the runtime's hydration win the race before the browser's lazy-load algorithm ever requests the placeholder value; verified the final rendered `<img>` still resolves to the correct cover image and the hero section is visually unchanged. No `support.js`/`image-slot.js` edit was needed or made.
### Added
- `favicon.ico` (root, multi-size 16/32/48/64) and `apple-touch-icon.png` (root, 180×180) — generated locally with Pillow from the existing `assets/favicon.png` (no new artwork, no placeholder/fabricated asset). Added `<link rel="apple-touch-icon" href="apple-touch-icon.png">` next to the existing `<link rel="icon">` on all 13 pages that have one, so browsers/iOS "Add to Home Screen" resolve a real icon instead of 404ing on the implicit default path.
- `scripts/qa-static-site.mjs` — dependency-free Node (ESM) static-asset/link checker, same style as `scripts/validate-content-data.mjs`. Scans every `*.dc.html` page and `index.html` for local `href`/`src` values and `<dc-import name="...">` references and confirms each resolves to a real file, case-sensitively (macOS is case-insensitive by default; GitHub Pages' Linux host is not — this is the same class of bug that produced the dead `SiteHeader.dc.html`/`SiteFooter.dc.html` files removed earlier). Skips dynamic `{{ }}` bindings, which `validate-content-data.mjs` already covers. Documented in `README.md` and `DEVELOPMENT.md`; added to `.github/pull_request_template.md`'s validation checklist.
- `.github/workflows/content-validation.yml`: new `static-site-qa` job runs `scripts/qa-static-site.mjs` alongside the existing content-data job, on the same PR/push-to-`main` triggers.
- `.github/workflows/pages.yml`: the `validate` job (which gates the Pages deploy) now also runs `scripts/qa-static-site.mjs`, so a broken static reference blocks deployment the same way a content-data error already does.
### Verified
- `node scripts/validate-content-data.mjs` — 0 structural errors (unchanged).
- `node scripts/qa-static-site.mjs` — 0 broken references across 117 checked local references on 16 pages; sanity-checked the script actually fails by temporarily introducing a case-mismatched path and confirming a non-zero exit, then reverting.
- `./scripts/preview.sh` + a full headless-Chromium sweep of all 14 site pages (plus the `index.html` redirect): zero console warnings/errors and zero network 404s on every page — down from the pre-existing single `{{ coverImage }}` 404 on each of the 3 guide pages.

# WearWyzer Mission Control — Ops Dashboard v1 (issue #19)

## Problem

Everything this repo's automation does — the engineering queue (issue #16), the guide
factory and deploy health/rollback (issue #17), the completion handoff watchdog (issue
#22), the OpenAI image renderer and spend ledger (issue #18) — is only visible by reading
GitHub issues, PRs, and workflow logs directly. There is no single place to answer "is
everything okay right now?" from a phone in under 10 seconds.

## Goal

A read-only, mobile-first status page that answers that question from a sanitized,
machine-readable snapshot — never by exposing raw issue/PR content, logs, or credentials.
Per the issue's product principle: **the dashboard observes automation; it does not
control merges, secrets, deployments, or publishing.** There are no buttons on this page
that mutate anything.

## What's implemented

- `scripts/ops-status-schema.mjs` — pure, dependency-free schema for the status
  artifact: enums for `overallHealth` (green/yellow/red), `automationState`
  (working/queued/review/blocked/failed/idle), CI/deployment/guide-factory/image-renderer
  states, plus `validateStatusShape()` (a **closed** shape — every nested object rejects
  unknown keys, not just checks required ones are present) and `findSecretLikeValues()`
  (recursive scan by credential-shaped key name and by value pattern — GitHub/OpenAI/Slack
  tokens, AWS keys, `Bearer` headers, PEM private keys). `computeStaleness()` is the shared
  timestamp-age check both the generator and the dashboard's own client-side logic are
  built on. Unit-tested in `scripts/__tests__/ops-status-schema.test.mjs`.
- `scripts/ops-status-builder.mjs` — pure assembly of the full status object from
  already-sanitized plain-data inputs (queue/issue state, status-log events, guide job
  manifests, the OpenAI spend ledger, deploy health). No I/O. Derives `automationState`
  from the active issue's labels (same label contract every other queue script uses —
  `docs/AUTOMATION_WORKFLOW.md`), `overallHealth` from a fixed precedence (incident or
  failed CI/deploy → red; blocked/failed automation, a stuck guide job, or an exhausted
  image budget → yellow; unknown CI/deploy data → yellow; otherwise green), and `blockers`
  from whatever of those is currently true. Unit-tested in
  `scripts/__tests__/ops-status-builder.test.mjs`, covering every health state, an empty
  queue, an active guide-factory job, failed CI, failed deployment, and incident
  suspension.
- `scripts/ops-status-cli.mjs` — the only file that touches disk or the network. Reads
  `automation/status/events.jsonl`, `automation/guide-jobs/*.json`,
  `automation/status/openai-spend.jsonl`, and `automation/status/last-healthy-deploy.json`
  locally; if `GITHUB_TOKEN`/`GITHUB_REPOSITORY` are set, also pulls live queue state
  (`ready`/`in-progress`/`blocked`/`site-incident` issues, the linked PR, and the latest
  `content-validation.yml` run on `main`) via the existing
  `scripts/queue-github-client.mjs` — no new secret. **Refuses to write `ops/status.json`
  if the assembled object fails `validateStatusShape()` or turns up anything in
  `findSecretLikeValues()`** — a build-time gate, not just a test-time one. Usage:
  `node scripts/ops-status-cli.mjs [--dry-run] [--now <iso>]`.
- `ops.dc.html` — the dashboard page itself. `<meta name="robots" content="noindex,
  nofollow">`, not linked from `Site Nav.dc.html`/`Site Footer.dc.html` or any other public
  page, and `robots.txt` additionally disallows `/ops.dc.html` and `/ops/` (defense in
  depth — the real access boundary is "nobody links to it," documented below). Fetches
  `./ops/status.json` on mount and every 60 seconds after (`cache: 'no-store'`, a
  cache-busting query param), computes staleness client-side from
  `generatedAtIso`/`staleAfterMinutes` the same way `computeStaleness()` does, and shows a
  distinct amber "data may be stale" banner instead of silently presenting old data as
  current. Mobile-first single-column card layout (no dense tables), a green/amber/red
  health hero card, and four expandable detail sections (Engineering, Content, Deployment,
  Incidents) so the 10-second read is the hero card and the detail is opt-in. Carries
  `apple-mobile-web-app-*` meta tags so "Add to Home Screen" on iOS behaves like a
  lightweight app shortcut — no service worker, no offline cache (see "Known v1
  simplifications").
- `ops/status.json` — a real, committed initial snapshot (generated by running the CLI
  once against this repo's actual state as of this change) so the dashboard has something
  to render and be verified against before the scheduled workflow is activated. Every
  subsequent run of `docs/automation/workflows/ops-status-refresh.yml` overwrites it.

## Status schema

See `scripts/ops-status-schema.mjs` for the authoritative, enforced shape. Summary:

| Field | Meaning |
|---|---|
| `schemaVersion` | Always `1` in this version. |
| `generatedAtIso` | When this snapshot was generated. |
| `overallHealth` | `green` \| `yellow` \| `red` — the 10-second answer. |
| `automationState` | `working` \| `queued` \| `review` \| `blocked` \| `failed` \| `idle`. |
| `activeWork` | The current `in-progress` issue (number/title/url) and its linked PR, or `null`. |
| `queue` | `{ depth, readyCount, blockedCount }` — counts only, no issue bodies. |
| `ci` | `{ status, lastRunIso, lastRunUrl }` for the latest `content-validation.yml` run on `main`. |
| `deployment` | `{ status, lastHealthyShaShort, lastCheckedIso }` — a 7-char SHA, never a full log. |
| `guideFactory` | `{ state, activeJobId, queuedCount }`. |
| `imageRenderer` | `{ state, monthlySpendUsd, monthlyCapUsd, budgetPct }` — spend totals only, never a per-call breakdown. |
| `incident` | `{ active, issueNumber, summary }` — `summary` is a truncated title, never the full incident report. |
| `blockers` | Short, human-readable summaries of what needs attention right now. |
| `lastMeaningfulActivityIso` | Latest of the last status-log event, active issue activity, and last deploy check. |
| `staleAfterMinutes` | How old `generatedAtIso` may get before the dashboard shows the stale banner (`30` by default). |

## Refresh cadence

The dashboard polls `ops/status.json` every 60 seconds. The generator
(`ops-status-refresh.yml`, staged) is scheduled every 15 minutes — conservative relative to
the client poll, since re-running it more often buys nothing until it's actually wired to
fire on the events it reports on (a future iteration). The dashboard's own staleness check
(`generatedAtIso` older than `staleAfterMinutes`, default 30) is what keeps "the generator
hasn't run in a while" visibly distinct from "the generator ran recently and everything is
idle" — the exact distinction issue #19 section 4 asks for.

## Why the status artifact is committed, not published some other way

Every other piece of automation state in this repo (`automation/status/*.jsonl`) is
intentionally **absent** from version control (`automation/status/README.md`) — it's
runtime-generated and only meaningful on the machine that produced it. `ops/status.json`
is different: it has to be a real file the deployed static site can `fetch()`, and this
repo has no backend and no build step (`CLAUDE.md`) to generate one at request time. The
only way for a statically-hosted `.dc.html` page to read fresh data is for that data to be
a committed file GitHub Pages serves. That's why `ops-status-refresh.yml` is the one staged
workflow in this repo with `contents: write` and a direct-to-`main` commit step, bypassing
the issue → PR → review flow every other automated change goes through — a deliberate,
narrow exception, scoped to a single generated JSON file that the CLI itself refuses to
write if it fails schema/secret validation first. A maintainer activating this workflow
should understand and accept that tradeoff (see the workflow file's own header comment).

## Local / unauthenticated runs

Running `node scripts/ops-status-cli.mjs` without `GITHUB_TOKEN`/`GITHUB_REPOSITORY` set
still produces a valid, schema-conformant status document — `queue` and `ci` degrade to
their "unavailable" shape (`automationState: "idle"`, `ci.status: "unknown"`) rather than
throwing. `overallHealth` becomes `yellow` in that case (an unknown CI/deploy state is
treated as worth a second look, not silently green) — this is expected and documented
behavior for a local dry run, not a bug.

## Limitations of unauthenticated static hosting (issue #19 section 6)

This dashboard has **no authentication**. Anyone who discovers the URL can view it — the
only protection is that it is not linked from anywhere public and carries `noindex,
nofollow` plus a `robots.txt` disallow, none of which are real access control (a
determined visitor, or anyone with repo access who already knows the URL, can always load
it). `ops/status.json` is designed to be safe to expose even so — no secrets, no raw
issue/PR/log content, only counts, enums, and short truncated summaries, enforced by
`findSecretLikeValues()` at generation time. **Real authentication would require a backend
or an access-controlled layer in front of GitHub Pages** (e.g. Cloudflare Access, a
signed-URL proxy, or moving the dashboard behind an authenticated deployment target) — none
of that exists in this repo today, and is explicitly out of scope for this issue's
"read-only, no control actions" product principle. Do not treat this page as a substitute
for real access control, and do not add anything to `ops/status.json` beyond what's already
enumerated in the schema table above without re-running the secret-scan reasoning in
`scripts/ops-status-schema.mjs`'s module header.

## Known v1 simplifications

- No service worker / offline cache — "PWA-style" here means the `apple-mobile-web-app-*`
  meta tags only (so "Add to Home Screen" looks like an app), not a true installable PWA
  with a manifest and offline support. Offline caching would also actively fight the
  staleness detection this dashboard depends on.
- `ci` only reflects `content-validation.yml` on `main` — not every workflow in this repo,
  and not per-PR CI state. A future iteration could expand this once there's a concrete
  need to see per-PR check status here.
- `queue.depth` currently equals `readyCount` (issues labeled `ready`, not yet dispatched).
  It does not include guide-factory `approved` jobs — those are surfaced separately under
  `guideFactory.queuedCount` since they're a different kind of work item.
- The generator is schedule-only in this version, not event-triggered — a stuck deploy or
  a newly opened incident is visible on the dashboard only once the next 15-minute run (or
  a manual `workflow_dispatch`) picks it up, bounded by the staleness banner in the
  meantime.

## Activation checklist (in addition to issue #16/#17/#18/#22's)

1. Read `docs/automation/workflows/ops-status-refresh.yml`'s header comment and confirm
   you're comfortable with its `contents: write` + direct-to-`main` commit step — this is a
   different permission/trust shape than every other staged workflow in this repo.
2. Copy `docs/automation/workflows/ops-status-refresh.yml` into `.github/workflows/`
   (same workflow-edit-permission constraint as every other staged workflow — Claude's
   GitHub App token cannot do this itself).
3. No new label is required.
4. Once active, confirm `ops.dc.html` on the live Pages deployment shows real (not just
   locally-generated) data after the first scheduled run.

```
node --test scripts/__tests__/*.test.mjs   # full suite, confirms no regression
node scripts/ops-status-cli.mjs --dry-run  # prints a generated status document without writing
node scripts/qa-static-site.mjs            # confirms ops.dc.html's local references resolve
```

Manual verification (this repo has no headless-browser CI step — see `CLAUDE.md`
"Verifying a change"): serve the repo with `./scripts/preview.sh`, open `/ops.dc.html`, and
confirm the dashboard renders with no `[dc-runtime]` unresolved-binding console warnings at
both a mobile (~375px) and desktop viewport width, that the health card updates on manual
refresh, and that the four detail sections expand/collapse.
